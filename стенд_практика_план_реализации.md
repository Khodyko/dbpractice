# Стенд для практической части: детальный план реализации

## 1. Цель стенда
- Подготовить полностью воспроизводимое окружение для демонстрации:
  - индексов и `EXPLAIN ANALYZE`;
  - партиционирования и `partition pruning`;
  - шардирования (targeted routing vs scatter-gather).
- Получить одинаковое поведение на любой машине за счет Docker.

## 2. Принципы демонстрации
- Все тезисы подтверждаются числами (`actual time`, `rows`, `buffers` при необходимости).
- Для каждого кейса есть формат:
  - **до** (неоптимальный вариант);
  - **после** (исправление);
  - краткий инженерный вывод.
- Не перегружаем блоки лишними операциями: фокус только на специфике механизма.
- Блок индексов: **один общий** Docker Compose и одна БД; SQL-кейсы идут по очереди (скрипты пересоздают таблицы).
- Партиционирование: один compose на блок (как индексы), SQL-скрипты по очереди; шардирование: **один общий compose** на весь блок + reset между кейсами.

## 3. Предлагаемая структура в репозитории
- `practice/README.md` — сценарий проведения практики и тайминг.
- `practice/postgres/` — PostgreSQL / Citus: Docker, SQL-скрипты
- `practice/mongo/` — MongoDB replica set + Spring Boot demo
- `practice/postgres/docker/` — compose-файлы PostgreSQL и `.env.example`
- `practice/postgres/sql/indexes/` — 6 SQL-файлов по индексным кейсам
- `practice/postgres/sql/partitioning/` — кейсы партиционирования
- `practice/postgres/sql/sharding/` — кейсы шардирования Citus
- `practice/postgres/sql/reset/` — очистка

## 4. Docker-окружение

## 4.1 Состав
- Индексы: один сервис `postgres` в `indexes.compose.yml` (БД `INDEX_DB`, порт `INDEX_PORT`).
- Партиционирование: один сервис `postgres` в `partitions.compose.yml` (БД `PART_DB`, порт `PART_PORT`); кейсы — отдельные SQL-файлы по очереди.
- Шардирование: один `sharding.compose.yml` с `citus-coordinator` и `citus-worker-01..03`; кейсы выполняются по очереди в одном кластере.
- Опционально UI (`adminer`/`pgadmin`) подключается только в тот compose, где нужен показ.

## 4.2 Требования к конфигурации
- Явно задать:
  - для индексов: `INDEX_DB`, `INDEX_PORT`, `POSTGRES_USER`, `POSTGRES_PASSWORD`;
  - для партиций: `PART_DB`, `PART_PORT`;
  - для шардовых кейсов: см. `practice/postgres/docker/.env.example`.
- Для воспроизводимости:
  - фиксировать версии образов;
  - вынести переменные в `.env.example`.

## 4.3 Команды запуска (runbook)

### Индексы
- Поднять `indexes.compose.yml`, дождаться healthcheck.
- По очереди выполнить `practice/postgres/sql/indexes/idx-0*.sql` (каждый скрипт самодостаточен, данные создаются через `INSERT` внутри скриптов).
- По завершении блока: `down -v`.

### Партиции
- Поднять `partitions.compose.yml`, дождаться healthcheck.
- По очереди выполнить `practice/postgres/sql/partitioning/part-0*.sql` (каждый скрипт самодостаточен).
- По завершении блока: `down -v`.

### Шарды
- Поднять `sharding.compose.yml`, дождаться readiness coordinator и worker-нод.
- По очереди выполнить SQL кейсов шардирования.
- Между кейсами запускать reset-скрипт (очистка схемы/таблиц), чтобы не смешивать эффекты измерений.
- По завершении блока: `down -v`.

## 5. Модель данных для демо
- Домен: `orders`/`events` (единый для всех блоков).
- Основные поля:
  - `id` (PK);
  - `tenant_id` (для шардирования);
  - `status` (для selectivity/partial index);
  - `created_at` (для range-фильтров и партиционирования);
  - дополнительные payload-поля для реалистичного размера строки.
- Распределение данных:
  - skew по `status` (чтобы показать низкую селективность);
  - равномерный/контролируемый поток дат;
  - достаточное число tenant-ов.

## 6. Блок 1: индексы и EXPLAIN ANALYZE

## 6.1 Обязательные 6 кейсов
1. Индекс подхватывается на селективном фильтре (`=` / узкий диапазон).
2. Индекс не выбирается при низкой селективности.
3. `Partial index`: совпадает/не совпадает предикат запроса.
4. Составной индекс: порядок столбцов влияет на план.
5. `Non-sargable` условие: функция/преобразование ломает индексный путь.
6. Неверные оценки planner из-за статистики; эффект `ANALYZE`.

## 6.2 Формат каждого кейса
- Шаг 1: SQL-запрос «до».
- Шаг 2: `EXPLAIN ANALYZE`.
- Шаг 3: изменение (индекс/перепись условия/`ANALYZE`).
- Шаг 4: `EXPLAIN ANALYZE` «после».
- Шаг 5: короткий вывод (1-2 инженерные фразы).

## 7. Блок 2: партиционирование

## 7.1 Что демонстрируем
- Сравнение `EXPLAIN ANALYZE` одного и того же запроса по диапазону дат: **без партиций** (полная таблица) и **с партициями и pruning**.
- Когда срабатывает `partition pruning`.
- Когда запрос идет по всем партициям.
- Поведение при «вне индекса»:
  - pruning есть, но внутри партиции `Seq Scan`;
  - pruning нет + `Seq Scan` по множеству партиций.

## 7.2 Практический сценарий
- Данные и объём строк задаются SQL-сценариями; партиционирование по `created_at` (месяц/квартал).
- Скрипт `part-00`: одна таблица `events_flat`, без партиций — бейзлайн времени и плана.
- Один и тот же запрос `count(*)` по марту:
  - в `part-00` — по `events_flat`;
  - в `part-01` — по `events_part` с партициями и корректным фильтром (pruning);
  - в `part-02` — условие, которое мешает pruning.

## 8. Блок 3: шардирование

## 8.1 Обязательные кейсы шардирования (согласовано)
1. **Без шардов vs с шардами (производительность)**:
   - один и тот же workload на локальной таблице PostgreSQL и на distributed-таблице Citus;
   - сравнение `EXPLAIN ANALYZE` и времени выполнения.
2. **HOT SPOT из-за плохого `shard key`**:
   - намеренно неудачный ключ распределения;
   - фиксация перекоса нагрузки и деградации latency.
3. **Ограничения `JOIN` distributed-таблиц**:
   - показать запрос, который не проходит/выполняется проблемно при некорректной модели распределения.
4. **Reference table (полное дублирование) как решение для `JOIN`**:
   - перевод малой таблицы в `reference table` (репликация на все воркеры);
   - повтор того же `JOIN` и демонстрация корректного выполнения.
5. **Co-location vs non-colocation**:
   - два варианта `JOIN` между distributed-таблицами;
   - объяснение, почему co-located сценарий выполняется корректно, а non-colocated приводит к ограничениям/деградации.

## 8.2 Опциональные кейсы (делаем при наличии времени)
- **Shard skew + rebalancing**:
  - контролируемо создаем перекос по шардам;
  - показываем восстановление баланса после ребаланса.
- **Router query vs distributed query**:
  - сравнение точечного запроса по `distribution key` и fan-out запроса без ключа.

## 8.3 Формат демонстрации шардирования
- Архитектурный показ + SQL proof-of-concept на каждом кейсе.
- Для каждого кейса: **до** → `EXPLAIN ANALYZE` → изменение модели/запроса → **после**.
- В стенде используем Citus-кластер и фиксируем единый набор метрик:
  - время выполнения;
  - ключевые узлы плана (`EXPLAIN VERBOSE`);
  - признаки fan-out/repartition и влияние на latency.

## 9. План прогона на выступлении (рекомендуемый)
1. Поднять `indexes.compose.yml`, последовательно выполнить все 6 SQL индексных кейсов, затем `down -v`.
2. Поднять `partitions.compose.yml`, последовательно выполнить SQL партиционирования (`part-00` … `part-03`), затем `down -v`.
3. Поднять один `sharding.compose.yml`, пройти 5 обязательных кейсов шардирования с reset между ними; при запасе времени — добавить 2 опциональных.
4. Финал: сводка trade-offs (ускорение чтения vs цена записи/эксплуатации).

## 10. Критерии готовности стенда
- Все команды выполняются по инструкции без ручных «догадок».
- Для каждого кейса есть предсказуемый план выполнения и понятный вывод.
- Результаты воспроизводимы минимум на двух запусках.
- Время полного прогона соответствует таймингу доклада.

## 11. Риски и как заранее снизить
- Разброс времени из-за «холодного» кэша:
  - фиксировать warm-up запросы.
- Нестабильные планы при изменении статистики:
  - явный `ANALYZE` перед нужными кейсами.
- Перегруз стенда лишними примерами:
  - держать только согласованные обязательные кейсы.
